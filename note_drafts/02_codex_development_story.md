# Codexと一緒に大学物理の問題集サイトを作った話

大学物理の問題を分野別・難易度別に探せるサイトを作りました。

https://nanana-2.github.io/physics-site/

この記事では，サイトの中身というより，どうやって作ったのかを書きます。

かなりざっくり言うと，MkDocsで静的サイトを作り，GitHub Pagesで公開し，Codexに実装や校正をかなり手伝ってもらいました。

個人的には，「AIに全部作らせた」というより，「自分が判断して，Codexに手を動かしてもらいながら，一緒に形にしていった」という感覚が近いです。

## 作ったもの

作ったのは，大学物理の問題集サイトです。

分野としては，

- 力学
- 電磁気学
- 熱力学
- 量子力学
- 統計力学

を扱っています。

問題ページはMarkdownで管理し，MkDocs MaterialでWebサイトとして表示しています。

分野別のサイドバー，難易度タグ，全問題一覧，参考文献ページ，SNSシェアボタン，プライバシーポリシーや利用条件ページなども入れました。

最初は単純なドキュメントサイトのつもりでしたが，作っているうちに「問題を探しやすいサイト」に寄せていった感じです。

## 技術構成

大まかな構成は次のような感じです。

- Markdownで問題文・解説を管理
- MkDocsで静的サイト化
- Material for MkDocsでデザイン
- GitHub Pagesで公開
- Google Analyticsでアクセス解析
- Search Consoleで検索インデックス確認
- Codexで実装・校正・整理を補助

技術的にものすごく凝った構成ではありません。

むしろ，静的サイトとしてなるべくシンプルに作っています。問題集サイトなので，ログイン機能やデータベースは不要です。Markdownを編集してビルドすれば公開できる，という構成の方が扱いやすいと思いました。

## Codexにやってもらったこと

Codexにはかなりいろいろなことを頼みました。

たとえば，

- Markdownファイルの整理
- カリキュラムに合わせた問題ファイルの移動
- 問題文・解説の校正
- MkDocsの設定修正
- トップページのデザイン調整
- 全問題一覧ページの実装
- スマホ向けの導線改善
- SNSシェアボタンの追加
- Google AnalyticsやSearch Console周りの設定
- プライバシーポリシーや利用条件ページの作成
- Amazonアソシエイトリンク周りの整理

などです。

この手の作業は，一つひとつはそこまで難しくなくても，量が多いとかなり大変です。

たとえば，「問題文の文末表現を整える」「解説だけ敬体にする」「過度な括弧表現を減らす」「試験問題っぽくない導入文を削る」みたいな作業は，人間だけでやるとかなり気が遠くなります。

でもCodexに指示を出すと，ファイルを見て，該当箇所を探して，修正して，ビルドして確認するところまでかなりまとめて進められます。

## 実際に出していた指示

指示は，かなり自然言語で出していました。

たとえば，

> 問題文は常体，解説は敬体にしてほしい。あくまでも文末の表現の問題なので内容そのものは変えない方針で。

とか，

> 統計力学の前半は，括弧を使った過度な補足表現や「」を使った強調表現が目立つから，試験問題っぽい自然な表現に直してほしい。

とか，

> 全問題一覧で，分野・章・難易度・キーワードから絞り込めるページを作りたい。

とか，

> ヒーローセクションの文字が大きすぎてAIっぽいから，もっと普通の文章くらいの大きさにしてほしい。

みたいな感じです。

細かい実装方法を全部指定するというより，「どういう状態にしたいか」「何は変えたくないか」を伝えることが多かったです。

特に大事だったのは，「内容そのものは変えない」「問題の要旨は維持する」「文体だけ直す」みたいな制約をはっきり書くことでした。

## Codexが便利だったところ

一番便利だったのは，ファイルをまたいだ修正です。

このサイトでは，問題ページがかなり多いので，普通に編集しているとファイル移動だけで疲れます。

Codexは，対象ファイルを探して，該当箇所を読んで，必要なところだけ直す，という作業が得意でした。

また，デザイン調整でもかなり役に立ちました。

トップページのヒーローセクションや，全問題一覧ページ，参考文献カード，スマホ表示などは，最初から完成形が見えていたわけではありません。

「なんかAIっぽい」「文字がでかい」「ピンクは違う気がする」「ネイビーをもう少しミッドナイトブルー寄りにしたい」みたいな雑な感想を投げて，それをCSSやHTMLの修正に落としてもらいました。

このあたりは，人間が方向性を判断して，AIに手を動かしてもらうのがかなり相性よかったです。

## 難しかったところ

もちろん，全部が一発でうまくいくわけではありません。

たとえば，デザインは何度もやり直しました。

最初は少し「AIで作りました」感が強く出てしまったので，文字を小さくしたり，装飾を減らしたり，色を落ち着かせたりしました。

また，問題文の校正でも，削りすぎると不親切になります。

大学物理の問題文は，背景説明を入れすぎると冗長になりますが，何も書かないと無機質になります。このバランスは，AIに任せきりにするより，人間が「これはいる」「これはいらない」と判断する必要がありました。

特に，統計力学や量子力学の問題では，壮大な言い回しや過度な導入文を削る作業が多かったです。

「幾何学的な関係」「数学的構造」「具体的に理解を深める」みたいな表現は，問題集としては少し大げさに見えることがあります。

## AIを使ううえで気をつけたこと

AIを使うと，作業量はかなり減ります。

ただ，内容の正しさまで丸投げするのは危ないです。

物理の問題では，式変形や前提条件が少しずれるだけで，問題として成立しなくなることがあります。

なので，Codexには主に次のような使い方をしました。

- ファイル整理を任せる
- 表現の校正を任せる
- UIの実装を任せる
- ビルド確認を任せる
- ただし問題の妥当性は自分でも見る

特に，問題文の内容を大きく変える修正は慎重にしました。

「文体だけ直す」「この式は変えない」「問題の意図は維持する」といった指示をかなり出していました。

## GitHub Pagesで公開するまで

サイトはGitHub Pagesで公開しています。

MkDocsでビルドして，`gh-pages` ブランチにデプロイする形です。

途中でGitHub認証やデプロイ周りで詰まったり，Google Analyticsがうまく認識されなかったり，Search Consoleの確認タグを入れたりしました。

このあたりも，Codexにターミナルのエラーを見せて，「これは何が原因か」「どこを直せばいいか」を相談しながら進めました。

エラー文をそのまま貼ると，原因の切り分けをしてくれるのはかなり助かりました。

## 完成したというより，公開できる状態になった

今のサイトは，完成したというより，公開できる状態になった，という感じです。

問題数が多いので，まだ誤りはあると思います。表現も直したいところが残っているかもしれません。

ただ，公開しないと検索にも載らないし，誰にも使われません。

なので，ある程度のところで公開して，見てもらいながら直していくことにしました。

個人サイトは，最初から完璧にするより，公開して育てる方が現実的だと思います。

## Codexを使ってみた感想

Codexは，単にコードを書く道具というより，「大量の細かい作業を一緒に進める相手」としてかなり便利でした。

特に今回のように，

- Markdownファイルが大量にある
- 文体をそろえたい
- サイトデザインも少しずつ調整したい
- ビルドやデプロイも確認したい

という作業では，相性がよかったです。

一方で，何を良しとするかは人間が決める必要があります。

「この表現は自然か」「このデザインは自分らしいか」「この問題は本当に必要か」みたいな判断は，やっぱり自分でやるところです。

AIに任せるというより，AIを横に置いて，自分の判断を高速に形にしていく感じでした。

## おわりに

大学物理の問題集サイトを作るつもりが，結果的に，Codexと一緒にかなり大きめの個人サイトを作る実験にもなりました。

作業量は多かったですが，思っていたより現実的に形になりました。

サイトはこちらです。
